VeryGood 자산 연동 실행 계획

목적

메인 허브(verygood-chocolate.com)의 문구/링크/썸네일/소개 카드를 실제 프로젝트 자산 기준으로 정리한다.

이번 단계에서는 외부 서비스 링크를 무리하게 배포하지 않고,
각 프로젝트 폴더를 직접 확인해서:
	•	소개 문구를 정리하고
	•	썸네일 또는 스크린샷을 만들고
	•	메인 허브의 Services / Digital / Store 섹션에 반영한다.

⸻

작업 원칙
	1.	한 번에 모든 폴더를 동시에 크게 수정하지 않는다.
	2.	먼저 자산 조사 → 소개 문구 정리 → 썸네일/스크린샷 확보 → 허브 반영 순서로 진행한다.
	3.	아직 배포되지 않은 앱/관리자/AI는 직접 링크 대신 소개 카드 중심으로 노출한다.
	4.	이미 배포된 서비스 또는 연결 가능한 페이지는 링크를 연결한다.
	5.	각 프로젝트는 우선 “읽고 분석”부터 하고, 바로 대공사하지 않는다.
	6.	섹션별 반영 자산 경로와 사용 이미지는 별도 mapping 파일로 정리한다.

⸻

현재 확인된 소스 폴더

1. Wellness App
	•	경로: john_2.0/code/welness_app
	•	상태: 앱이라 외부 링크 없음
	•	처리 방향:
	•	코드/화면을 확인해 소개 문구 작성
	•	가능한 경우 대표 화면 스크린샷 또는 목업용 이미지 확보
	•	메인 허브의 Digital > Wellness App 카드에 반영
	•	링크 대신 소개 페이지 또는 프리뷰 카드로 노출

2. Wellness Admin
	•	경로: john_2.0/code/welness_admin
	•	상태: 아직 배포 전
	•	처리 방향:
	•	구조/화면 확인
	•	소개 문구 작성
	•	스크린샷 또는 대표 화면 비주얼 확보
	•	메인 허브의 Digital > Admin Dashboard 카드에 반영
	•	소개 중심 노출, 실제 로그인/사용 링크는 숨김

3. AI Beta
	•	상태: 아직 미구현
	•	처리 방향:
	•	지금은 실제 기능 링크 없음
	•	Digital > AI Recommendation Beta 카드만 유지
	•	문구는 “베타 준비 중 / 추천 실험 중” 톤으로 처리
	•	실제 링크는 넣지 않음

4. Cake Reservation
	•	경로: code/partners
	•	상태: 소스 존재
	•	처리 방향:
	•	프로젝트 확인 후 소개 문구 정리
	•	대표 이미지/스크린샷 확보
	•	Services > Cake Reservation 카드와 상세 소개에 반영
	•	실제 배포 경로가 있으면 연결

5. Review Page
	•	경로: code/영수증리뷰이벤트
	•	상태: 소스 존재
	•	처리 방향:
	•	프로젝트 확인 후 소개 문구 정리
	•	대표 이미지/스크린샷 확보
	•	Services > Reviews 카드에 반영
	•	실제 리뷰 페이지 링크가 있으면 연결

6. Store / Experience Page
	•	경로: code/vcc체험단페이지
	•	상태: 소스 존재
	•	처리 방향:
	•	매장/체험 페이지 성격 분석
	•	메인 허브의 Store 섹션 또는 Visit the Store 비주얼 보강에 활용
	•	소개 문구와 대표 이미지 반영

Codex 작업 순서

Step 1. 조사

각 폴더에 대해 아래를 확인한다.
	•	앱/웹의 목적
	•	주요 화면 구성
	•	대표적으로 보여줄 수 있는 화면
	•	메인 허브에 넣을 때 필요한 한 줄 설명
	•	공개 링크 존재 여부

Step 2. 자산 수집

각 프로젝트에서 가능한 경우 아래를 확보한다.
	•	대표 스크린샷 1~2장
	•	썸네일로 쓸 수 있는 화면 캡처
	•	이미지가 없으면 코드 구조를 바탕으로 카드형 임시 비주얼 제작

Step 3. 문구 작성

각 프로젝트별로 아래를 정리한다.
	•	카드 제목
	•	카드 설명 1문장
	•	상세 소개 2~3문장
	•	버튼 문구
	•	링크 방식(실링크 / 소개만 / 준비중)

Step 4. 허브 반영

메인 허브의 섹션에 반영한다.
	•	Digital > Wellness App
	•	Digital > Admin Dashboard
	•	Digital > AI Beta
	•	Services > Cake Reservation
	•	Services > Reviews
	•	Store 또는 Visit the Store

Step 5. 정리 문서화

반영 후 아래를 별도 정리한다.
	•	사용한 폴더 경로
	•	추출한 이미지/스크린샷 경로
	•	카드별 연결 방식
	•	아직 미확정인 링크 목록

메인 허브 반영 규칙

	•	링크 대신 소개 카드 중심
	•	“기록과 추천을 연결하는 웰니스 앱” 톤 유지

Wellness Admin
	•	내부 운영용이라는 점이 드러나야 함
	•	직접 로그인 유도보다 “운영을 위한 관리 시스템” 소개 중심

	•	“Beta Preview” 또는 “Coming Later” 톤 가능

Cake Reservation

	•	링크가 있으면 연결 우선

Reviews
	•	고객 반응과 매장 경험을 보여주는 카드
	•	실제 후기 흐름이 잘 보이도록

Store / Experience
	•	오프라인 경험과 매장 분위기 강화용
	•	사진/비주얼 품질이 중요

이번 단계에서 하지 않을 것

	•	아직 배포되지 않은 기능을 실제 서비스처럼 과장 노출
	•	AI Beta를 메인 핵심 CTA로 올리기
	•	조사 없이 이미지와 문구를 임의 추정으로 확정하기

Codex에게 바로 줄 실행 지시

아래 순서로 진행한다.
	1.	john_2.0/code/welness_app 읽고 목적/화면/대표 비주얼 후보를 정리
	2.	john_2.0/code/welness_admin 읽고 목적/화면/대표 비주얼 후보를 정리
	3.	code/partners 읽고 Cake Reservation 카드용 자산 정리
	4.	code/영수증리뷰이벤트 읽고 Reviews 카드용 자산 정리
	5.	code/vcc체험단페이지 읽고 Store 섹션용 자산 정리
	6.	각 항목별 카드 문구와 링크 방식 제안
	7.	마지막에 메인 허브 반영안 정리

중요:
	•	한 번에 전부 수정하지 말고, 위 순서대로 조사 → 제안 → 반영한다.
	•	이미지가 없으면 무리하게 찾지 말고, 스크린샷/임시 비주얼 제작 방향을 제안한다.
	•	각 프로젝트별 결과를 먼저 보여준 뒤 반영에 들어간다.
